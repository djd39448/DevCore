# ADR-0004 — Cookbook image storage: Supabase Storage + on-device LRU cache

This ADR resolves workload-spec §6.4. The Conductor captured this decision at
the `behavior_spec` human gate; the Architect does not re-litigate it.

Depends on the workload spec, the behavior spec §2.4 *cookbook_recipes*,
§3.5–§3.6 *cookbook surfaces*, §4.8 *image generation*, and §6.3 *cookbook
save and image-prompt regeneration*.

---

## Context

The web app stores `recipeImagePrompt` (a text string) for each cookbook
recipe and regenerates the image bytes from the prompt **every time the
recipe is viewed**. The image is held in the DOM as a `data:image/png;base64`
URL. The maintainer's own notes describe this as a deliberate trade-off to
avoid bloating the database with bytes (`replit.md` and behavior spec §2.4).

The trade-off is bad on mobile:

- Every cookbook tap incurs a 2–10 second image-generation latency.
- Every regeneration costs an `gpt-image-1` API call.
- A switch between iPhone and (future) iPad regenerates everything from
  scratch.
- Cellular data usage is unpredictable — every image is a fresh download.

Options weighed at the gate:

- **Supabase Storage + on-device LRU cache** — Generate once on save,
  PUT to a Supabase Storage bucket, store the URL on the cookbook row;
  iOS caches the most-recent ~N MB locally.
- **On-device only with eviction** — Cache the generated bytes locally,
  regenerate if evicted. Cross-device cookbook browsing pays the
  generation cost again.
- **Status quo** — Regenerate every view. Bad UX, expensive.
- **Pre-generate at save time, no device cache** — Simplest, but every
  view is a network fetch.

## Decision

**Generate the image once at cookbook save time, store the bytes in
Supabase Storage, store the public URL on the `cookbook_recipes` row, and
maintain an LRU image cache on the iOS device.**

- Bucket: `cookbook-images` (one Supabase Storage bucket, RLS-scoped per
  user).
- Object key: `${user_id}/${cookbook_recipe_id}.png`.
- The `cookbook_recipes` row carries `image_url` (the public Storage URL)
  and `image_prompt` (kept for regeneration if the user explicitly asks).
- iOS uses `NSCache`-backed disk LRU sized at **128 MB** (≈ 100–200
  cookbook images at typical PNG sizes). Eviction is LRU by access time.

If the user explicitly taps a "regenerate image" action on a cookbook
recipe, the backend issues a fresh generation, overwrites the Storage
object, and the iOS app evicts the cached entry.

## Status

**Accepted.** Locked by Dave at the behavior_spec gate, 2026-05-24.

## Consequences

### Positive

- Cookbook open is **instant** after the first view.
- AI generation cost happens **once per saved recipe**, not per view.
- Cross-device browsing works without re-generation, because the URL is
  shared state on Supabase Storage.
- Cellular data usage is one PNG download per cookbook view (vs. a fresh
  generation round-trip).

### Negative

- Storage cost: ~ $0.021/GB/month on Supabase Storage. At 1 MB per
  recipe, 1000 recipes = $0.02/month. Negligible.
- The first cookbook save now incurs an image-generation latency at save
  time, not at first view. Mitigation: the save action is a deliberate
  user moment ("save this recipe"), where a 2–5 second wait is acceptable
  with a spinner.
- An RLS policy must scope the Storage bucket per user. Standard Supabase
  pattern; small policy file.

### What this implies for the contract

- The `cookbook_recipes` row gains an `image_url TEXT` column.
- The `image_prompt` (today: `recipeImagePrompt`) column stays — it is
  the regeneration seed.
- The `thumbnailUrl` field in the current source is **dropped** —
  `image_url` subsumes it.
- The contract adds:
  - `POST /api/kitchen/cookbook` generates the image inline before
    returning the new row.
  - `POST /api/kitchen/cookbook/:id/regenerate-image` issues a fresh
    generation, overwrites the Storage object, returns the (same) URL
    with a cache-busting `?v=<timestamp>` query string.
- The contract does **not** specify the iOS cache size or eviction
  policy — that's an iOS-track implementation choice. The behavior spec's
  guarantee is "the URL works on each call; the image bytes at the URL
  are stable until a regeneration is requested".
