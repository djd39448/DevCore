# ADR-0003 — Auth: Sign in with Apple + Supabase email/OTP

This ADR resolves workload-spec §6.3. The Conductor captured this decision at
the `behavior_spec` human gate; the Architect does not re-litigate it.

Depends on the workload spec and the behavior spec §7 *Auth flow*.

---

## Context

The web app uses **Replit Auth via OpenID Connect + Passport.js** with a
PostgreSQL session table (`connect-pg-simple`). All `/api/kitchen/*` routes
are gated by an `isAuthenticated` middleware. This mechanism is platform
lock-in and is replaced wholesale (workload-spec §5).

What survives the port (behavior spec §7.2):
- a stable user identity per logged-in human;
- a session lifetime measured in days, not minutes;
- protected backend routes gated on identity;
- user data upserted on login.

Options weighed at the gate:

- **SIWA only** — One credential. Required by the App Store anyway. If
  the user's Apple ID is locked, the app has no recovery path.
- **SIWA + Supabase email/OTP** — Two methods. OTP is six-digit code sent
  to email. Supabase ships both natively. No passwords stored.
- **SIWA + magic link** — Email click-a-link instead of type-a-code.
  Adds friction on mobile (switch apps and back).
- **SIWA + email/password** — Traditional. Adds password storage, reset
  flow, breach handling. Not worth the cost on a personal app.

## Decision

**Sign in with Apple is the primary method; Supabase email + one-time code
is the fallback.** No passwords. No magic links.

Both methods land the user into the **same Supabase Auth user record** —
email is the identity key. SIWA's "hide my email" relay address is
acceptable; Supabase treats the relay address as the user's email.

## Status

**Accepted.** Locked by Dave at the behavior_spec gate, 2026-05-24.

## Consequences

### Positive

- App Store compliance is satisfied: SIWA is offered whenever any other
  third-party login is offered. (We never had any; this just future-proofs
  if a third-party provider is added later.)
- One Supabase user table is the source of truth. The Go backend sees a
  single `user_id` regardless of how the human logged in.
- Email/OTP gives a recovery path that does not depend on Apple's ID
  service.
- No passwords means no breach-impact, no reset endpoints, no rate-limit
  arms race on a login form.

### Negative

- Two flows to design in the iOS app's first screen. Mitigation: the
  Supabase iOS SDK ships UI helpers; both flows are 5–10 lines each.
- SIWA's email-relay address can change if the user revokes Apple
  permission. Mitigation: identity is keyed on Supabase user-id, not on
  email; email is a contact, not a primary key.

### What this implies for the contract

- The backend never sees the SIWA token directly. The iOS app exchanges
  the Apple ID token with Supabase Auth; Supabase returns a JWT; the iOS
  app sends the JWT as `Authorization: Bearer <jwt>` to the Go API. The
  Go API verifies the JWT against Supabase's JWKS endpoint.
- The contract specifies: every `/api/kitchen/*` endpoint requires a valid
  Supabase JWT. The Go middleware extracts `user_id` from the JWT
  `sub` claim.
- The `users` table in the new schema is the Supabase Auth user record;
  the Go backend treats it as read-only. No upsert-on-login flow.
- The contract does **not** specify a logout endpoint — token expiry plus
  Supabase's `signOut` SDK handles it client-side.
